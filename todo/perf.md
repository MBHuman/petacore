# Сводка по профилированию и рекомендации

Коротко — выводы по недавно собранным профилям (`cpu.pprof`, `heap.pprof`, `allocs.pprof`, `trace.out`):

- Наблюдается сильный рост аллокаций (GB) в рантайме при выполнении запросов.
- Основные аллоцирующие узлы:
  - `petacore/internal/runtime/planner.performNestedLoopJoin`
  - парсер/rsql/ANTLR (`parser.*`, `rsql/visitor`, `antlr4`)
  - исполнитель/планировщик (`executor.ExecuteStatement`, `planner.ExecutePlanNode`)
- Нет ярко выраженного lock-contention (mutex/block профили пустые).
- Trace показывает значительную работу GC (gcDrain, gcBgMarkWorker) и I/O (netpoll/syscall.write).

Рекомендуемые улучшения (быстрые победы → средние → сложные):

1) Быстрые (низкий риск, быстрый эффект)
  - Переиспользование буферов: заменить повторные `make([]byte, ...)` и временные строки/буферы на переиспользуемые `bytes.Buffer` или `[]byte` из `sync.Pool`.
  - Ввести `sync.Pool` для часто создаваемых парсерных/контекстных структур (ANTLR contexts, AST nodes) и для промежуточных объектов планировщика.
  - Включить `-benchmem` и `allocs` при локальных bench для отслеживания эффекта изменений.

2) Средние (меньше аллокаций, архитектурные простые правки)
  - Минимизировать копирование данных в `performNestedLoopJoin` (передавать ссылки, использовать итераторы вместо временных срезов).
  - Перейти на потоковую обработку результатов (streaming) где возможно, чтобы уменьшить удержание больших буферов.
  - Сократить промежуточные структуры, которые живут дольше запроса и удерживают GC объём.

3) Глубокие (алгоритмические улучшения)
  - Пересмотреть реализацию `performNestedLoopJoin` (альтернативные алгоритмы, хеш-джойн, шардирование данных).
  - Оптимизировать парсер/lexer (меньше выделений при разборе, reuse token streams) или кешировать разобранные плановые шаблоны для повторяющихся запросов.

Диагностика и валидация

- Команды для быстрого воспроизведения и профилирования (пример):
```
go build -o pcore ./cmd/pcore
./pcore --pprof --pprof-addr=localhost:6060 --store=inmemory
curl 'http://localhost:6060/debug/pprof/profile?seconds=20' -o cpu.pprof
curl 'http://localhost:6060/debug/pprof/heap' -o heap.pprof
curl 'http://localhost:6060/debug/pprof/allocs' -o allocs.pprof
go tool pprof -http=:8080 ./pcore cpu.pprof
```

- Сравнение heap до/после патча: снять `heap.pprof` до и после и использовать `pprof` для diff.

План работ (короткий)

1. `Investigate performNestedLoopJoin` — получить `pprof -list` и найти строки с аллокациями.
2. Внедрить `sync.Pool` для парсерных контекстов и буферов; прогон bench с `-benchmem`.
3. Оптимизировать копирование в `performNestedLoopJoin` и повторно измерить.
4. По результатам — рассмотреть алгоритмическую переработку join.

Файлы с профилями: положите `cpu.pprof`, `heap.pprof`, `allocs.pprof`, `trace.out` в корень репозитория для дальнейшего анализа.
