# PetaCore Runtime Architecture

This document describes the high-level architecture of the PetaCore SQL runtime engine, organized around a separation of concerns principle similar to DuckDB's design.

## Overview

The PetaCore runtime is organized into several key modules that handle different aspects of SQL query processing:

```
┌─────────────┐
│  SQL Input  │
└──────┬──────┘
       │
       ▼
┌─────────────────┐
│  Parser/Lexer   │
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  Semantic Layer │
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  Planner        │
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  Executor       │
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  Storage Engine │
└─────────────────┘
```

## Core Modules

### 1. **Parser** (`./parser/`)
Converts SQL text into an Abstract Syntax Tree (AST).

- **Location**: `internal/runtime/parser/`
- **Generated by**: ANTLR4 from `sql.g4` grammar
- **Components**:
  - `sqlLexer.go` - Tokenization
  - `sqlParser.go` - Parsing rules
  - `sql_base_listener.go` - Base visitor implementation

**See**: [Parser Documentation](./Parser.md)

### 2. **Query Language & Items** (`./rsql/`)
Intermediate representation layer that provides semantic structures for SQL constructs.

- **Location**: `internal/runtime/rsql/`
- **Submodules**:
  - `statements/` - SQL statement definitions
  - `items/` - Expression and clause definitions
  - `table/` - Result set representation
  - `visitor/` - AST visitor for conversion to semantic layer

### 3. **Query Planner** (`./planner/`)
Converts a SELECT statement into a tree of logical plan nodes that represent the execution strategy.

- **Location**: `internal/runtime/planner/`
- **Key Components**:
  - `planner.go` - Plan creation from SELECT statements
  - `plan.go` - Plan node definitions
  - `executor.go` - Plan execution engine
  - `set_operations.go` - UNION/INTERSECT/EXCEPT handling

**Features**:
- Modular plan node architecture
- Support for complex queries (JOINs, subqueries, aggregations)
- Set operations (UNION, INTERSECT, EXCEPT)

**See**: [Planner Documentation](./Planner.md)

### 4. **Query Executor** (`./planner/executor.go` & `./executor/`)
Executes logical plan nodes against the storage engine.

- **Location**: `internal/runtime/planner/executor.go` and `internal/runtime/executor/`
- **Key Responsibilities**:
  - Recursive plan traversal
  - Tuple processing and filtering
  - Expression evaluation
  - DDL operations (CREATE TABLE, DROP TABLE, etc.)

**See**: [Executor Documentation](./Executor.md)

### 5. **Function System** (`./functions/`)
Extensible system for SQL functions and operators.

- **Location**: `internal/runtime/functions/`
- **Features**:
  - Built-in functions (aggregate, scalar, etc.)
  - User-defined functions
  - Type coercion

**See**: [Functions Documentation](./Functions.md)

### 6. **Helper Modules** (`./rhelpers/`)
Runtime helper functions for expression evaluation and type conversion.

- **Components**:
  - `parser.go` - Operator parsing and evaluation
  - `rmodels/` - Runtime models
  - `subquery/` - Subquery execution
  - `revaluate/` - Expression re-evaluation
  - `rops/` - Runtime operations

### 7. **Wire Protocol** (`./wire/`)
PostgreSQL-compatible wire protocol for client communication.

- **Location**: `internal/runtime/wire/`
- **Protocol**: PostgreSQL Frontend/Backend Protocol v3.0

**See**: [Wire Protocol Documentation](./Wire.md)

### 8. **System Tables** (`./system/`)
Implementation of PostgreSQL-compatible system catalog tables.

- **Location**: `internal/runtime/system/`
- **Tables**: `pg_class`, `pg_attribute`, `pg_namespace`, etc.

## Execution Flow

### Query Execution Pipeline

```
1. SQL Text
   ↓
2. Lexer/Parser → AST
   ↓
3. Visitor Pattern → Semantic Layer (statements, items)
   ↓
4. Planner → LogicalPlan (tree of PlanNodes)
   ↓
5. Executor → ExecuteResult
   ↓
6. Wire Protocol → Client Response
```

### Plan Nodes (Logical Operators)

The planner generates a tree of logical plan nodes:

- **ScanPlanNode** - Table scan
- **ProjectPlanNode** - Column projection
- **FilterPlanNode** - WHERE clause filtering
- **JoinPlanNode** - Table joins
- **AggregatePlanNode** - GROUP BY aggregation
- **SortPlanNode** - ORDER BY sorting
- **LimitPlanNode** - LIMIT/OFFSET
- **UnionPlanNode** - UNION operation
- **IntersectPlanNode** - INTERSECT operation
- **ExceptPlanNode** - EXCEPT operation (set difference)
- **SubqueryPlanNode** - Subqueries in FROM
- **ValuesPlanNode** - SELECT without FROM

## Transaction Model

- **Isolation Level**: Snapshot Isolation
- **Concurrency Control**: MVCC (Multi-Version Concurrency Control)
- **Transaction Support**: 
  - Automatic transactions for single statements
  - Custom context support for transaction boundaries

## Key Design Principles

1. **Separation of Concerns**: Parser, planner, and executor are distinct phases
2. **Tree-Based Execution**: Plans and results are tree structures
3. **Composability**: Plan nodes can be nested to express complex queries
4. **Extensibility**: Easy to add new plan node types and operations
5. **Type Safety**: Go's type system used extensively
6. **Memory Safety**: MVCC ensures transactional consistency

## Runtime Helpers

The `rhelpers` package provides utilities for runtime expression evaluation:

- **Expression Models** (`rmodels/`) - Runtime representation of SQL expressions
- **Type Conversion** - Automatic type coercion for operations
- **Operator Evaluation** - Binary and unary operators
- **Subquery Execution** - Nested query support

## Storage Integration

The runtime abstracts storage through a clean interface:

```go
interface DistributedStorageVClock {
    Read()          // Tuple retrieval
    Write()         // Tuple insertion  
    Update()        // Tuple modification
    Delete()        // Tuple deletion
    GetTable()      // Table metadata
    CreateTable()   // Schema creation
}
```

## Performance Considerations

1. **Stream Processing**: Executor uses streaming where possible
2. **Lazy Evaluation**: Subqueries evaluated only when needed
3. **Short-Circuit Evaluation**: Boolean expressions evaluate early exit
4. **Memory Efficiency**: MVCC avoids full table copies

## Extension Points

1. **New Operators**: Add plan node types in `planner/plan.go`
2. **New Functions**: Register with function registry
3. **Custom Visitors**: Extend `rsql/visitor/`
4. **Storage Backends**: Implement storage interface
